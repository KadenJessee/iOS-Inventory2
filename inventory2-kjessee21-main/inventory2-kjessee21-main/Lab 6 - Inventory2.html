Introduction

The objective of this Lab is for you to demonstrate your understanding of persistence. We will build upon the original Inventory assignment and add one important feature: reading and writing to and from a SQLite3 database. You should have completed the study of "Swift SQLite Tutorial for Beginners" and completed the Inventory assignment before attempting this assignment. You will continuing to edit the original Inventory project, but you will instruct Xcode to push to the new inventory2 repo on github.com.

If you completed Inventory, you know what it will look like. The only difference is that if you add items, then "Swipe Up" the Inventory from the app switcher (try it), when you restart Inventory the added items will be gone. However, your goal with Inventory2 is those added items are saved to a database when the app goes either inactive or into the background, and your items are read into an empty items array and displayed when the app again becomes active.

The app will look to the user absolutely identical to the original Inventory app, with no additional UI, popups or screens.

Enhance the Inventory app

These are the general requirements for the Inventory2 app:

  • If the app is about to be moved into the background by changing to the inactive state, your app will save the array of Items to a SQLite3 database.

  • If the program is terminated (by first tapping HOME, then double-tapping the home button and swiping the Inventory2 app upward), the database should contain all inventory items from the table view, preserving the order.

  • When the user restarts the app, the app reloads the Items array from the database, again, preserving the order.

Specific instructions

The detection of a change in app state and the reading or writing of the database is attached to any child view of any currenly visible at the time the app is put into the background. The way I did this in my version of this assignment is I created two new functions, readDatabase() and writeDatabase().

writeDatabase() saves the records to the database in response to a notification that the app is about to become inactive.

readDatabase() clears the items array, then reads the database and populates the items array in the order they were in the items array when they were written into the database. In other words, do not sort or reorder the items.

To detect this app status change, add the .onChange modifier to one of your child views in each of your three views, the main view, the edit view and the add view:

struct ContentView: View {
    @Environment(\.scenePhase) var scenePhase
    @EnvironmentObject var arr: MyStuff
                    .
                    .
                    .
    .onChange(of: scenePhase) { newPhase in
        if newPhase == .active {
            readDatabase(items: &arr.items)
        } else if newPhase == .inactive {
            writeDatabase(items: &arr.items)
        }
    }

In my version of this lab, I created arr, a class instance that contains my array of items and added it to the environment in InventoryApp.swift:

@main
struct InventoryApp: App {
    @StateObject private var arr = MyStuff()

    var body: some Scene {
        WindowGroup {
            ContentView()
                .environmentObject(arr)

        }
    }
}

You might be wondering just what is &arr.items. The & tells Swift to send a reference to arr.items to the function rather than a copy of it. arr is the name of an instance of class (I named my class MyStuff) that contains an array of items (I put this code in MyStuff.swift):

import Foundation
import SwiftUI

struct Item: Hashable, Identifiable {
    var id = UUID()
    var sDesc = ""
    var lDesc = ""
}

class MyStuff: ObservableObject {
    @Published var items = [Item]()

Suggested logic of readDatabase(items: inout [Item])

  • delete every entry in items to prepare to read them from the database

  • To use SQLite3 functions, add #import SQLite3 near the top of the .swift file(s) containing SQLite3 calls.

  • Create the database as documented in the module reading material. I suggest the filename (or path component) "Inventory.sqlite".

    • Create a SQLite3 table called Items if it does not already exist, with 3 fields (the id field is there to keep the items in order):

    • id: INTEGER PRIMARY KEY AUTOINCREMENT

    • shortDescription: VARCHAR

    • longDescription: VARCHAR

    • Create a SELECT statement that will retrieve each item, initialize an Item object and append the Item to the Items array.

  • Create the writeDatabase() function outside of any view or in its own .swift file with the following signature:

    func writeDatabase(items: inout [Item]) { ... }

Suggested logic of writeDatabase(items: inout [Item])

  • open the SQLite3 database

  • execute the SQL statement "DELETE FROM Items" to clear the database

  • for each item in the items[] array (e.g. for item in items {…​ }):

  • call sqlite3_prepare() to prepare the INSERT SQL statement string

  • call sqlite3_bind_text() to bind the shortDescription and longDescription fields. When specifying these text fields as the third parameter of sqlite3_bind_text(), code them as:

    (....Description as NSString).utf8String

  • call sqlite3_step() to execute the "INSERT INTO Items" to add each item to the database Items table.

  • close the database with sqlite3_close()
